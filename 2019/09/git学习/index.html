Git学习

九月 15, 2019 发布在 category, subcategory

Git

  • 分布式版本控制系统

    SVN与Git的主要区别

  • SVN：集中是式版本控制系统，版本库是集中放在中央服务器的，尔干活的时候用自己电脑，所以首先要从中央>服务器那里得到最新版本，修改完成后自己电脑推送到中央服务器

  • Git是分布式控制系统，它没有中央服务器，每个人的电脑就是一个完整的版本库，工作时候不需要联网。

    • 需要填写一个用户名和邮箱作为标识
    • git config –global user.name “xxx”

    • git config –global user.email “xxx”

      创建版本库

  • makdir test

  • cd test

  • git init

    • 把目录变成git可以管理的仓库，目录下由隐藏文件.git

    • ls -a 命令行现实所有文件

      添加文件到版本库,修改文件

  • touch readme.txt 创建文件

  • git add readme.txt 添加到暂存区

    • 没有任何提示说明添加成功

  • git commit -m “readme.txt提交”

  • git status 查看是否还有文件未提交

    • 查看有没有modify到 文件

  • 修改文件（视频音频二进制文件无法追踪）

    • git status 追踪变动的文件

  • git diff readme.txt 追踪查看变动内容

    • cat readme.txt 明朗化一次显示整个文件

  • 修改完成后继续提交

    • 重复开始2步骤（git add；git commit -m）

      版本退回

  • git log 查看文件操作记录

    • 版本号，提交时候的注释
    • git log –pretty=oneline （简略显示）

  • git reset –hard HEAD^ 退回上个版本 ^^上上个版本

    • git reset –hard HEAD~100退回100个版本

  • 退回版本操作之后git log无法看到最新的版本纪录

    返回最新版本

  • git reflog 查看所有版本号来进行恢复 git reset –hard 版本号

    理解工作区与暂存区的区别

  • 工作区 的文件，目录文件

  • 版本库（repository）：工作区的隐藏目录文件.git

    • 包含暂存区（stage）
    • 自动创建的第一个分支master，以及指向master的一个指针HEAD

  • 提交文件的两个步骤

    • git add 把文件添加到暂存区

    • git commit 提交更改，把暂存区的内容提交到当前分支上

      撤销修改

  • 已经git add，git commit提交了，退回版本

  • 已经git add，修改以后重新add

  • git checkout –..“撤销工作区的修改”

    • “–”很重要，不然就变成创建分支了

删除文件

  • 文件git add，git commit提交之后，删除文件的话
    • rm ，删除stage中，没有git add 操作，git checkout – 看英文提示
    • git rm 删除并git add 操作，git reset撤销操作，看英文提示
  • 命令行删除以后可以有2个操作
    • 从版本库恢复
    • git commit -m “” 删除版本库内容

  • git checkou –

    • 没有git commit 之前撤销操作

      远程仓库

  • ssh key

    • ssh-keygen -t rsa –C “youremail@example.com”
    • id-rsa是私钥，不能泄露出去，id-rsa.pub是公钥，可以放心地告诉任何人

  • 创建github 账号，新建repository，根据提示把本地建立的库和远程关联

    • git remote add origin https://github.com/gmTcover/ForMac.git

  • git push -u origin master

  • 由于远程库是空的，我们第一次推送master分支时，加上了 –u参数，Git不但会把本地的master分支内容推送的远程新的master分支，还会把本地的master分支和远程的master分支关联起来，在以后的推送或者拉取时就可以简化命令。推送成功后，可以立刻在github页面中看到远程库的内容已经和本地一模一样了

    通过操作本地库，可以修改远程库

  • 本地文件的修改提交

  • 删除本地库中的文件在提交（本地和远程是关联的）

    • git push origin master

      创建与合并分支

  • git checkout -b dev

    • 创建并（-b）表示切换到该分支
    • 相当于git branch dev；git checkout dev两个步骤

  • git branch

    • 查看当前分支

  • git checkout master

    • 切换到主分支

  • 在主分支下合并分支内容修改的内容

    • git merge dev 快速合并
    • Fast-forward模式

  • 合并完成后删除分支

    • git branch -d dev

      总结

  • 查看分支：git branch

  • 创建分支：git branch name

  • 切换分支：git checkout name

  • 创建+切换分支：git checkout –b name

  • 合并某分支到当前分支：git merge name

  • 删除分支：git branch –d name

如何解决分支冲突

  • 对同一个文件，主分支，和创建分子上都做了修改
    • both modified name
    • cat name 查看不同内容

  • 修改分支内容和master相同，解决冲突（该过程无法切换分支，看英文解释）

    • 之后git add，git commit

      分支管理策略。

  • 通常合并分支时，git一般使用”Fast forward”模式，在这种模式下，删除分支后，会丢掉分支信息，现在我们来使用带参数 –no-ff来禁用”Fast forward”模式。

  • 创建一个dev分支。

  • 修改readme.txt内容。

  • 添加到暂存区。

  • 切换回主分支(master)。

  • 合并dev分支，使用命令

    • git merge –no-ff -m “注释” dev

  • 查看历史记录

    • git log
    • 图形显示分支合并的commit

  • git log –graph –pretty=oneline –abbrev-commit

  • git log –pretty=oneline

    处理多个分支

  • 隐藏当前工作区 git stash

    • 开始新建分支完全其他任务
    • 在开发中，会经常碰到bug问题，那么有了bug就需要修复，在Git中，分支是很强大的，每个bug都可以通过一个临时分支来修复，修复完成后，合并分支，然后将临时的分支删除掉。比如我在开发中接到一个404 bug时候，我们可以创建一个404分支来修复它，但是，当前的dev分支上的工作还没有提交

  • git stash list

  • git stash apply恢复，恢复后，stash内容并不删除，你需要使用命令git stash drop来删除。

  • git stash pop,恢复的同时删除stash内容也删除

    • stash 内容，之前工作的被存储在了一个地方。可以删除可以不删除

      多人协作

  • 远程repository

    • git remote
    • git remote -v

  • 多人协作工作模式一般是这样的

  • 首先，可以试图用git push origin branch-name推送自己的修改.

  • 如果推送失败，则因为远程分支比你的本地更新早，需要先用git pull试图合并。

    • git pull 是这和先提交的人合并（但是需要你的修改，不是别人的，同时比你先提交，你的无法提交）
    • git pull也失败了
    • 看英文解释原因是没有指定本地dev分支与远程origin/dev分支的链接，根据提示，设置dev和origin/dev的链接，如下
    • git branch –set-upstream name origin/name,然后在git pull，抓取同名试图merge
    • 但是可能会产生冲突
    • 手动修改冲突和上面相同 cat 看冲突内容，修改之后git add；git commit -m “”
    • 最后git push origin dev

  • 如果合并有冲突，则需要解决冲突，并在本地提交。再用git push origin branch-name推送

  • git branch –set-upstream-to=origin/

    • 指定本地dev分支与远程origin/dev分支的链接

      返回版本操作

  • git relog

  • git reset –hard HEAD^
